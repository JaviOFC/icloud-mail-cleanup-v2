       **Pipeline execution:**
       - `@work(exclusive=True, thread=True) def run_pipeline(self)`:
         - Get db_path from `self.app.db_path`
         - If db_path is None, try default path discovery (same as CLI's `open_db(None)` behavior)

         **Step 1: Scan**
         - `self.call_from_thread(log.log_step, "Step 1/3: Scanning Envelope Index...")`
         - Import and call `open_db(db_path)`, `scan_messages(conn)`
         - Also load: `get_sent_recipients`, `get_replied_conversation_ids`, `get_sender_stats`, `get_sender_display_names`, `get_document_attachment_message_ids`
         - Check `worker.is_cancelled` after each heavy call
         - `self.call_from_thread(progress.update, advance=1)`
         - Log message count

         **Step 2: Classify**
         - `self.call_from_thread(log.log_step, "Step 2/3: Classifying messages...")`
         - Build contact profiles via `build_contact_profiles()`, run `classify_messages()`
         - Save checkpoint via `save_checkpoint()`
         - `self.call_from_thread(progress.update, advance=1)`
         - Log classification tier distribution

         **Step 3: Analyze (content)**
         - `self.call_from_thread(log.log_step, "Step 3/3: Content analysis...")`
         - This step is expensive (MLX embeddings). Log that it may take several minutes.
         - Replicate the core logic from `cmd_analyze()`: load checkpoint, parse .emlx files, generate embeddings, cluster, fuse classifications, save merged checkpoint
         - Since cmd_analyze() uses ProcessPoolExecutor internally, the thread worker orchestrates while MLX runs in subprocess
         - `self.call_from_thread(progress.update, advance=1)`

         **Completion:**
         - Reload app data: re-read checkpoint, rebuild report_data, update `self.app.classifications`, `self.app.report_data`, `self.app.messages`
         - `self.call_from_thread(log.log_success, f"Pipeline complete: {count} emails classified")`
         - Update status to "Pipeline: Complete"
         - Show notification

       - `on_button_pressed(event)`:
         - #btn-pipeline: disable button, enable cancel, run `self.run_pipeline()`
         - #btn-cancel: cancel worker, re-enable pipeline button

       **Important:** The pipeline replicates CLI logic but within Textual's worker framework. It imports scanner, contacts, classifier, checkpoint, embedder, clusterer modules directly -- NOT cmd_* functions (those assume argparse namespace). Extract the core pipeline steps.
